product overview

產品定位與設計挑戰

這是一個結合內建錢包與 DEX 聚合器的 Web3 工具。對比 MetaMask 純錢包與 Uniswap 純 DEX 的設計，這個產品兼具兩者 — 用戶可以在同一個 App 內管理私鑰、簽名交易、執行 Swap、瀏覽 DApp。最大的挑戰是用戶體驗與安全的權衡。

挑戰 01私鑰本地化的 UX 成本

私鑰永遠不離開用戶裝置是 Web3 原則 — 但這意味著每一次簽名都要解密私鑰、用戶要記得密碼、助記詞要妥善保存。最終採用 AES-256-GCM 加密 + scrypt KDF (N=262144)，平衡安全性與解鎖速度。

挑戰 02多鏈 RPC 的可靠性

Alchemy/Infura 任一節點掛掉用戶就無法交易。設計 FallbackProvider 自動切換主備節點，搭配每條鏈獨立的 RPC 池，並監控延遲與成功率動態調權。

挑戰 03Approve 風險的教育成本

無限 Approve (MaxUint256) 是 Web3 詐騙最大來源。產品端強制預設為精確授權 + 提供「授權管理」頁面讓用戶撤銷舊授權。MetaMask 都做了之後我們抄了。

關鍵決策 #1
內建錢包 + 外部錢包雙模式

不只做內建錢包 — 同時接 MetaMask + WalletConnect。讓重度用戶可以用習慣的錢包，新手也可以快速上手內建版。

關鍵決策 #2
用 wagmi/viem 而非 ethers.js

wagmi 已是 React DApp 的事實標準 (連 Uniswap 都換了)。生態整合好、Hook 設計優雅、TypeScript 友好。但底層仍可呼叫 ethers/viem 補低階操作。

關鍵決策 #3
Auto Router vs 自建路徑搜尋

MVP 用 Uniswap 官方 Auto Router (省 4 週開發 + 路由品質高)。Phase 2 再評估自建多 DEX 聚合 (Uniswap + Curve + 1inch)。

關鍵決策 #4
精確 Approve vs 無限 Approve

強制預設精確授權。會多一次 Approve TX (用戶體驗稍差) 但能避免 99% 的 Rug Pull/詐騙風險。差的體驗 vs 用戶資金損失 — 後者代價太高。

core features

核心功能與技術細節

2.1內建錢包 (Wallet Core)

採用 BIP39/BIP32/BIP44 標準，路徑 m/44'/60'/0'/0/x 支援多帳號派生。所有私鑰使用 AES-256-GCM 加密儲存於本機 Secure Storage (iOS Keychain / Android EncryptedSharedPreferences / Web IndexedDB + WebCrypto)。

// 助記詞生成 (BIP39)
const mnemonic = ethers.Mnemonic.entropyToPhrase(ethers.randomBytes(16)); // 12 words
const wallet = ethers.HDNodeWallet.fromPhrase(mnemonic, undefined, "m/44'/60'/0'/0/0");

// 加密存儲
const encryptedJson = await wallet.encrypt(userPassword); // scrypt KDF
// 存入 IndexedDB（Web）或 Keychain（App）

// 解密還原
const decryptedWallet = await ethers.Wallet.fromEncryptedJson(encryptedJson, userPassword);

2.2外部錢包連接 (wagmi + viem)

// wagmi config — 統一管理所有 Connector
export const config = createConfig({

});

2.3Swap 核心流程

// 1. 檢查 Allowance
const allowance = await tokenContract.allowance(userAddress, ROUTER_ADDRESS);

// 2. 如需 Approve（精確授權）
if (allowance < amountIn) {
  await tokenContract.approve(ROUTER_ADDRESS, amountIn);  // ⚠️ 不用 MaxUint256
}

// 3. 執行 Swap
const swapTx = await routerContract.swapExactTokensForTokens(
  amountIn,
  amountOutMin,  // 扣除滑點
  [tokenA, tokenB],
  userAddress,
  deadline
);

5.3合約風險偵測

風險偵測方式用戶提示
假 Token對比 Token List 白名單🟡 此 Token 未經驗證
Rug Pull檢查合約是否含 pause() / blacklist()🔴 合約含可暫停交易函式

5.4Approve 無限授權風險

// ❌ 危險：無限授權
await token.approve(router, ethers.MaxUint256);

// ✅ 安全：精確授權
await token.approve(router, exactAmountNeeded);

// 一鍵撤銷授權
async function revokeApproval(tokenAddr, spenderAddr, signer) {
  const token = new ethers.Contract(tokenAddr, ERC20_ABI, signer);
  return token.approve(spenderAddr, 0); // 設為 0
}

UI 設計：Swap 時預設精確授權；提供「授權管理」頁面列出所有已授權 Token + Spender；每項顯示「撤銷」按鈕。

5.5防釣魚機制

啟動時載入 MetaMask 釣魚黑名單 (auto-updated)。DApp 連線前檢查 domain：blacklist 直接阻擋、whitelist 直接通過、unknown 顯示警告。所有外部連結強制顯示完整 URL，可疑 URL 高亮提醒。

6.3不要做（重要）

  • 自建 AMM 合約 — 直接用 Uniswap，不要重造輪子
  • 自建 Token List — 用 Uniswap 官方 Token List
  • 自建 RPC 節點 — Alchemy 免費方案足夠
  • Limit Order — 複雜度高，MVP 不需要
  • 跨鏈橋 — 複雜度極高，Phase 3+
  • 流動性提供 (Add Liquidity) — MVP 只做 Swap
