product story

動手寫 PRD 之前，我先想清楚的事

這不是一個「拉新活動」 — 是一個用 100K USDT 預算精準打三種用戶的成長引擎。下方五個區塊是我做這個產品時真正交付的東西。

🎯
01 · Problem & Why

為什麼這是「給 B 端辦活動的引擎」？

📌 一個重要的定位

這不是某個平台「自己辦」的活動 — 是給 B 端客戶辦活動用的引擎。CEX 與 DEX 客戶都能掛載。我們提供活動架構、反作弊、業績計算與排行榜後端，客戶決定獎池大小、活動規則、發放邏輯。廣告怎麼打、用戶從哪來、CAC 多少 — 那是 B 端客戶自己的業務，我們不過問也算不到。

客戶上線交易所或錢包後，下一個焦慮：新用戶從哪來？老用戶怎麼留？ 純靠 KOL 發單帶量成本高、生命週期短；發空投又容易養出撸毛黨；單做一個大型獎勵活動，預算很快被頭部用戶吃光，散戶完全沒感覺。

客戶試過的方法都有問題：

  • 單一獎勵池：100% 預算發 Top10，前 10 名搶得頭破血流，91% 的用戶完全沒參與感
  • 純空投：拉來的是撸毛黨，發完就走，留存接近 0
  • KOL 給佣金：佣金太高燒不起、太低 KOL 不上心，且 KOL 帶來的用戶質量參差

所以我把活動設計成「三個互不衝突的子模組 + 同一個業績計算引擎 + 統一反作弊」的套件。客戶可以全裝、可以只裝其中一兩個，獎池大小完全自訂：

  • 社群報銷（參考設計 500 USDT × N 人）→ 打 KOL，要素材審核確認真實性
  • VIP 挑戰賽（參考設計最高 30K USDT）→ 打核心節點，分期釋放鎖長期承諾
  • 每日排行榜（參考設計 5K USDT/日）→ 打活躍用戶，創造每天回來的理由

客戶簽約後，我會帶他做一場「活動設計 workshop」 — 根據他的預算、用戶結構、市場習慣，調出他要的三模組組合與參數。同一套引擎，對 CEX 客戶跑交易量排行、對 DEX 客戶跑 TVL 排行，業績計算規則參數化切換即可。

02 · Target Users

三種用戶 → 三個活動精準對應

傳統做法是「一個活動讓所有人都來玩」 — 結果三種人都覺得不適合自己。我選擇相反路線：三種用戶 × 三個活動 × 三套激勵邏輯。

📢
KOL / 社群領袖
有線下號召力，能舉辦實體活動 / 帶單群。動機：取得平台官方背書 + 報銷預算。
→ 社群報銷
💼
核心節點 / VIP 玩家
已有 VIP3+ 等級，手上有大客戶基礎。動機：衝級獎金 + 把已有資源轉化成業績。
→ VIP 挑戰賽
⚡
活躍散戶 / 競爭型用戶
每天都登入、想看到自己在榜上、追求每日成就感。動機：每日榜單名次 + 即時獎勵。
→ 每日排行榜

我學到的事：拉新活動不是「越大越好」，是「越精準越好」。一個 100K 大池吸引不到 KOL（覺得自己拿不到頭獎），也吸引不到散戶（覺得跟自己無關）。三個池子各打一群人，每群人都覺得「這個活動是為我設計的」。

05 · MVP Hypothesis

想驗證的三個假設

H1 · 三活動分流假設

三個活動互不衝突 — 同一個用戶只參與「對自己最有利」的那個，不會三個都搶。

驗證方法：活動結束後分析數據，>70% 用戶集中參與一個活動。若三個活動用戶高度重疊代表分流失敗。

H2 · 留存機制有效假設

分期釋放（40/30/30）+ 業績下滑追回機制，能讓拉來的用戶留至少 60 天。

驗證方法：VIP 挑戰賽參與者 60 天後留存 > 50%，業績下滑 ≥50% 追回案例 < 10%。

H3 · 反作弊夠強假設

四維反作弊（設備指紋 / IP / KYC / 行為模式）能攔住 95% 以上的刷帳號。

驗證方法：活動結束抽樣 1000 帳號人工複核，異常帳號漏報率 < 5%。

06 · My Product Decisions

我做過的關鍵產品決策

我的決策 #1
三活動分流 vs 一個大活動

單一大活動是「省事」的方案，但用戶分群差異太大不能用同一套激勵。我選擇拆成三個獨立活動 — 開發成本高 1.5 倍，但用戶覆蓋率從 ~20% 拉到 ~80%，預算 ROI 更高。決策關鍵：開發成本 vs 用戶覆蓋率，後者更值得投資。

我的決策 #2
排行榜每 5 分鐘刷新而非即時

即時刷新最有臨場感，但後端壓力暴增 + 高頻 query 容易被刷。每小時刷新沒有緊張感，用戶不會頻繁回來看。5 分鐘是測試過 1/5/15/60 分鐘四個選項後的結論 — 既有競爭感，後端壓力又可控。

我的決策 #3
VIP 挑戰分 3 期釋放，不一次性給

一次性發放：用戶達標立刻贖回離場，活動結束 TVL 暴跌。分 3 期（40 / 30 / 30）+ 業績下滑追回：強迫用戶持續維持業績才能拿完獎金。這是「鎖長期承諾」的設計，比「直接發錢」對平台健康度更好。

我的決策 #4
社群報銷強制人工審核（即使慢）

其他兩個活動都是系統自動發獎。社群報銷不行 — 必須人工審核活動真實性。AI 生成的活動影片、PS 的合照、湊人頭照片，自動審核擋不住。寧可發獎慢 24 小時，也不能讓平台預算被假活動掏空。這是「對的事情 > 體驗順暢」的選擇。

附錄 · Technical Challenges

技術挑戰與工程協作

以下技術細節是我與工程團隊共同設計 — 我負責優先順序、跨團隊協調與業務需求對齊；架構與實作細節由工程架構師主導。

挑戰 01：排行榜 5 分鐘刷新的 QPS 控管

所有用戶幾乎同時在 5 分鐘整點刷新，後端壓力暴增。設計快取層（Redis sorted set）+ 排行榜計算 cron job 每 5 分鐘預算一次寫入 Redis，前端讀取 Redis 而非實時計算。

挑戰 02：分期釋放的時間鎖機制

VIP 挑戰賽獎金 40/30/30 分 3 期，每期釋放前要檢查當下業績是否符合條件。設計 settlement_schedule 表記錄每筆獎金的釋放條件與檢查時間，cron job 每日掃描觸發釋放或追回。

挑戰 03：四維反作弊判定

同 IP + 同設備指紋 + 同 KYC 證件號 + 行為模式（註冊時間間隔 / 操作軌跡）四維度聯合判定。任一維度命中需人工複核，三維度命中自動凍結獎勵。

anti-fraud

風控與防作弊

4.1防自刷機制

維度規則觸發動作
設備指紋同一 device_fingerprint 註冊多帳號僅保留首帳號，其餘不計入新增
IP 地址同 IP 24h 內註冊 ≥ 3 帳號標記異常，人工審核
KYC同一身份證/護照號碼僅首帳號有效，重複註冊拒絕
行為模式批量註冊時間間隔 < 30 秒自動標記為機器人，全部不計入

4.2有效用戶定義

  • 註冊 + 通過 KYC + 首次有效投資 ≥ 100 USDT
  • 首筆投資需存續 ≥ 7 天，否則不計入
  • 註冊後 14 天內無任何操作，自動標記為無效
  • 系統偵測同設備/同IP/同KYC 的關聯帳號

4.3排行榜防刷

規則觸發條件處理方式
異常波動單日業績突增 ≥ 500%自動凍結排名，人工覆核
短時衝量1 小時內集中大額入金標記觀察，延遲計入
集中提現上榜後 48h 內提現 ≥ 80%追回當期獎勵
多帳號協作多帳號向同一上級集中投資合併計算，異常帳號凍結

4.4報銷審核規則

審核項目合格標準不合格處理
影片真實性可辨識活動場地、參與者、時間退回 + 允許補件 1 次
照片數量≥ 3 張不同角度退回 + 允許補件
品牌露出可見品牌 LOGO / 名稱退回 + 允許補件
參與人數照片/影片可辨識 ≥ 5 名參與者退回
素材時效活動日期需在活動期間內不合格，不可補件
素材原創AI 偵測 + 人工比對永久取消資格

4.5異常處理分級

Level 1 輕微：單項異常指標 → 標記觀察，不影響正常使用。

Level 2 中度：多項異常或單項嚴重 → 凍結相關活動獎勵，發送人工覆核工單。

Level 3 嚴重：確認作弊或利用漏洞 → 取消活動資格 + 追回已發獎勵 + 帳號降級。

Level 4 封號：重複嚴重違規或涉及金融詐騙 → 永久封禁帳號 + 凍結所有資金。

4.6技術實作

前端：
- React / Vue3 + Vite
- TailwindCSS
- 排行榜：Polling (5min interval)
- 倒數計時：Server Time Sync 避免客戶端時間篡改

後端：
- RESTful API + Rate Limiting
- 排行榜計算：定時任務每 5 分鐘跑一次 SQL 聚合
- 快照機制：23:59:59 鎖定排名寫入 daily_ranking
- 風控引擎：獨立微服務，事件流即時判斷
- 獎勵發放：排程任務 + 異步處理 + 冪等保護
